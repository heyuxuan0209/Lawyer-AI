# 法律客户咨询助手产品方案

更新时间：2026-05-19

## 1. 项目背景

本项目的目标是做一个面向律师和律所的法律场景 Agent 产品。前期讨论过 `claude-for-legal`、AlphaGPT，以及律师日常工作中的 5 类高频场景：

1. 撰写文件：按文库模板生成法律文书，后续人工排版、查错、查重。
2. 信息获取：每日获取法律行业动态、新政策、新判例，并快速阅读。
3. 文案宣传：基于热点生成文章、图片、视频，用于自媒体发布。
4. 客户咨询：寒暄、客户提出诉求、律师给出法律定性与解决方案，分免费和付费咨询。
5. 案件代理案头工作：文书 OCR、资料输入、初步分析、法条/判例优化、律师确认。

经过比较，第一阶段建议优先切入 **客户咨询助手 / 谈案助手**。

核心判断：

> 不是做一个替律师直接回答客户的法律问答机器人，而是做一个帮助律师完成咨询记录、案情结构化、初步法律分析、待补材料清单、服务方案和法律意见书草稿的工作助手。

核心链路：

```text
咨询记录
-> 案情结构化
-> 初步法律分析
-> 待补材料清单
-> 服务方案 / 法律意见书草稿
-> 律师确认
```

## 2. 为什么先做客户咨询

### 2.1 离成交最近

客户咨询是律师业务入口。律师在咨询中要快速理解事实、判断法律关系、识别证据缺口、决定是否承接、解释服务边界和报价。

如果 AI 能把一次咨询自动沉淀成案情摘要、争议焦点、法律关系、待补材料清单、初步方案和法律意见书草稿，律师能马上感知价值。

### 2.2 输入天然存在

客户咨询本来就会产生语音、微信聊天记录、会议纪要、客户上传材料、律师手写备注。这些都可以成为 Agent 的输入。

相比每日法律资讯、自媒体文案，客户咨询更贴近律师核心工作，不需要先建设一个完整大而全的法律知识库才能启动。

### 2.3 风险更可控

产品定位不是让 AI 直接对客户给最终法律意见，而是让 AI 辅助律师：

- 记录咨询过程
- 整理案情事实
- 提醒追问问题
- 生成初步分析草稿
- 生成服务方案草稿
- 交由律师确认后再对外发送

这样比直接做面向客户的法律问答机器人更安全，也比完整案件代理流程更容易落地。

### 2.4 能自然延展

客户咨询之后，可以继续延展到：

- 法律意见书
- 律师函
- 起诉状 / 答辩状
- 案件材料清单
- 报价方案
- 委托合同
- 后续案件代理工作台

因此客户咨询是入口，后续可以串起文书撰写和案件代理。

## 3. 对 claude-for-legal 的借鉴

`claude-for-legal` 值得借鉴的不是具体法律内容，而是 Agent 产品架构和工作流方法。

### 3.1 按业务场景拆 Agent

不要做一个万能法律助手，而应该按咨询类型拆分：

- 婚姻家事咨询
- 劳动争议咨询
- 合同纠纷咨询
- 民间借贷咨询
- 公司股权咨询
- 刑事初询
- 交通事故 / 人身损害咨询
- 常年法律顾问咨询

每个咨询类型的事实结构、追问问题、证据材料、风险提示、交付件格式都不同。

### 3.2 每个 Agent 有自己的 playbook

`claude-for-legal` 的关键机制是让 Agent 学习团队的工作规则。对应到本项目，应沉淀“中国律师咨询 playbook”：

- 哪些信息必须先问清楚
- 哪些内容可以免费初步判断
- 哪些内容必须转为付费咨询
- 哪些风险不能直接承诺
- 如何引导客户补充事实
- 如何判断是否值得接案
- 如何生成服务方案
- 咨询纪要采用什么格式
- 对客户可见内容和律师内部判断如何区分

### 3.3 Human-in-the-loop

所有输出都应定位为“律师复核草稿”，不是最终法律意见。

产品需要明确：

- AI 可以辅助整理和生成草稿
- 法律结论必须由律师确认
- 对客户发送前必须经过律师审核
- 高风险结论必须标注需进一步核验

### 3.4 引用与核验机制

法律 Agent 的底线是不能伪造法律依据。产品应要求：

- 每个法律结论尽量绑定法条、案例或检索来源
- 法条需标明是否现行有效
- 案例需记录案号、法院、裁判日期、裁判要旨
- 没查到来源时明确标记“需进一步核验”
- 不允许把模型记忆当作现行法律依据

## 4. 对 AlphaGPT 的借鉴

AlphaGPT 更接近中国律师真实使用场景，值得借鉴的是功能组合和交付件形态。

### 4.1 AI 辅助咨询

可借鉴为客户咨询中的三层输出：

1. 给客户听得懂的初步结论
2. 给律师看的法律依据和风险点
3. 给后续办理用的行动建议

### 4.2 AI 法律意见

客户咨询结束后，可自动生成：

- 《咨询纪要》
- 《初步法律分析意见》
- 《法律服务方案》
- 《待补材料清单》

如果客户付费或委托，再进一步生成更正式的《法律意见书》。

### 4.3 AI 法律阅读

客户咨询常伴随材料上传，例如合同、协议、欠条、劳动合同、判决书、聊天记录、转账记录等。

产品应支持：

- 上传材料
- 自动摘要
- 提取关键事实
- 与咨询记录交叉验证
- 标注矛盾点
- 标注缺失证据

### 4.4 类案检索报告

客户经常会问“这个案子胜算怎么样”。产品不应直接输出确定性胜诉率，而应辅助律师生成：

- 相似案例方向
- 法院裁判倾向
- 支持或不支持的关键事实
- 影响结果的证据因素
- 需要进一步检索和核验的问题

## 5. 产品差异化

如果只做“法律问答 + 文书生成”，会很像 AlphaGPT，也很难形成差异。当前产品应把差异放在 **咨询现场辅助 + 咨询后成交转化 + 办案延展**。

### 5.1 从工具变成咨询现场助手

AlphaGPT 更像律师主动输入问题后使用的工具。本产品应贴近“谈案现场”：

- 实时转写客户表达
- 自动识别案由和关键信息
- 提醒律师继续追问
- 标记客户表达里的矛盾
- 提醒风险话术
- 自动生成咨询纪要
- 咨询结束后生成服务方案

### 5.2 从回答问题变成帮助承接案件

客户咨询不是纯法律研究，也有业务转化目的。产品应辅助律师判断：

- 客户是否有明确诉求
- 事实是否初步清楚
- 证据是否初步充分
- 标的额和代理费是否匹配
- 是否值得承接
- 是否需要转付费咨询
- 如何解释服务边界
- 如何生成报价方案
- 如何生成跟进任务

### 5.3 从单次咨询变成咨询到办案链路

目标链路：

```text
客户咨询
-> 自动纪要
-> 案情结构化
-> 待补材料清单
-> 初步法律分析
-> 服务方案 / 报价
-> 委托后转案件工作台
-> 生成文书 / 证据目录 / 时间线
```

Claude for Legal 偏工作流插件，AlphaGPT 偏法律 AI 工具。本项目的机会是做成：

> 咨询入口 + 案件转化 + 后续办案的律师业务助手。

## 6. MVP 定义

第一版不要做大而全，优先做一个能在真实咨询后交付结果的产品。

### 6.1 输入

- 咨询录音或转写文本
- 微信聊天记录
- 客户上传材料
- 律师手动补充备注
- 咨询类型选择，如劳动争议、民间借贷、婚姻家事、合同纠纷

### 6.2 核心输出

第一版至少输出四类内容：

1. 咨询纪要
2. 案情摘要
3. 风险与争议焦点
4. 下一步行动清单

增强输出：

5. 待补材料清单
6. 初步法律分析意见
7. 服务方案 / 报价建议草稿
8. 法律意见书初稿

### 6.3 MVP 工作流

```text
创建咨询
-> 选择咨询类型
-> 上传录音 / 聊天记录 / 材料
-> AI 生成结构化案情
-> AI 提醒缺失问题
-> 律师补充或修改
-> AI 生成初步分析和服务方案
-> 律师确认
-> 导出 / 发送给客户 / 转案件
```

## 7. 页面与功能结构

### 7.1 咨询列表页

用于管理所有咨询线索：

- 客户姓名
- 咨询类型
- 咨询状态
- 最近更新时间
- 是否已生成纪要
- 是否已生成服务方案
- 是否转委托
- 跟进提醒

建议状态：

- 待整理
- 待补材料
- 待律师确认
- 已发送方案
- 已转委托
- 暂不承接

### 7.2 单个咨询工作台

建议采用三栏结构：

左侧：输入与材料

- 咨询录音 / 转写
- 聊天记录
- 上传材料
- 律师备注
- 客户基础信息

中间：案情结构化

- 客户诉求
- 关键事实
- 时间线
- 人物关系
- 证据材料
- 缺失信息
- 矛盾点
- 可能案由

右侧：AI 辅助与交付件

- 追问建议
- 风险提示
- 法律关系初判
- 争议焦点
- 待补材料清单
- 服务方案草稿
- 法律意见书草稿
- 一键转案件

### 7.3 律师确认机制

所有对外内容必须经过律师确认：

- AI 生成内容默认是草稿
- 律师可以逐段确认、修改、删除
- 高风险内容需要显著标记
- 对外版本和内部版本分开
- 保留修改记录，便于复盘

## 8. 优先咨询场景

第一阶段建议先选 2-3 个高频、标准化、材料相对清晰的场景。

### 8.1 劳动争议

适合原因：

- 高频
- 事实结构相对清晰
- 证据类型明确
- 客户诉求直接

典型材料：

- 劳动合同
- 工资流水
- 社保记录
- 解除通知
- 考勤记录
- 微信沟通记录

典型输出：

- 是否存在劳动关系
- 解除是否合法
- 可主张项目
- 证据缺口
- 仲裁路径
- 服务方案

### 8.2 民间借贷

适合原因：

- 咨询量大
- 材料类型明确
- 容易形成问诊表

典型材料：

- 借条 / 欠条
- 转账记录
- 聊天记录
- 催收记录
- 对方身份信息

典型输出：

- 借贷关系是否初步成立
- 诉讼主体是否明确
- 利息是否合法
- 证据是否足够
- 管辖和时效问题
- 起诉材料清单

### 8.3 婚姻家事

适合原因：

- 高频
- 客户付费意愿相对明确
- 咨询过程需要大量事实梳理

典型材料：

- 结婚证
- 房产信息
- 银行流水
- 子女信息
- 债务材料
- 家暴或出轨证据

典型输出：

- 离婚路径
- 财产线索
- 抚养权争议点
- 证据缺口
- 调解 / 诉讼建议
- 服务方案

## 9. Agent 角色设计

第一版可以不用做复杂多 Agent，但内部能力可以按角色拆分，便于后续扩展。

### 9.1 咨询记录员

负责：

- 转写整理
- 去除口语噪音
- 保留关键原话
- 生成咨询纪要

### 9.2 案情结构化助手

负责：

- 提取客户诉求
- 提取事实
- 生成人物关系
- 生成时间线
- 识别证据材料
- 标注缺失信息

### 9.3 法律初判助手

负责：

- 判断可能法律关系
- 提示可能案由
- 提示争议焦点
- 标注法律风险
- 标注需核验依据

### 9.4 材料清单助手

负责：

- 根据场景生成待补材料清单
- 区分必要材料和辅助材料
- 生成客户可理解的材料说明

### 9.5 服务方案助手

负责：

- 生成服务路径
- 生成报价说明草稿
- 生成客户跟进任务
- 判断是否建议承接

## 10. 风险与边界

### 10.1 产品边界

产品不是：

- 面向客户的自动法律咨询机器人
- 自动给出最终法律意见的系统
- 自动承诺胜诉率的系统
- 替代律师判断的系统

产品是：

- 律师咨询过程助手
- 案情结构化工具
- 法律分析草稿生成工具
- 服务方案生成工具
- 咨询到办案的转化工具

### 10.2 法律风险控制

必须避免：

- 伪造法条
- 伪造案例
- 给出确定性胜诉承诺
- 忽略时效、管辖、证据等关键问题
- 把未经律师确认的内容直接发送客户

建议机制：

- 所有法律结论默认标为“待律师确认”
- 高风险结论增加红色标记
- 法条和案例引用必须可追溯
- 无依据时标注“需进一步检索核验”
- 客户可见内容与律师内部分析分离

## 11. 迭代路线

### Phase 1：咨询后整理

目标：解决律师咨询后整理成本高的问题。

功能：

- 上传咨询文本 / 录音转写文本
- 自动生成咨询纪要
- 自动生成案情摘要
- 自动生成待补材料清单
- 自动生成下一步行动清单

验证指标：

- 单次咨询整理时间减少
- 律师修改成本可接受
- 输出结构稳定
- 律师愿意在真实咨询中使用

### Phase 2：咨询中辅助

目标：进入谈案现场。

功能：

- 实时转写
- 实时提取关键信息
- 实时追问提醒
- 风险话术提示
- 咨询结束后自动生成交付件

验证指标：

- 律师是否愿意现场打开使用
- 追问建议是否准确
- 是否减少遗漏问题
- 是否提升咨询专业感

### Phase 3：服务方案与转委托

目标：从咨询工具升级为业务转化工具。

功能：

- 生成服务方案
- 生成报价说明
- 生成委托前材料清单
- 生成客户跟进任务
- 咨询一键转案件

验证指标：

- 咨询到付费转化是否提升
- 跟进是否更及时
- 客户是否更容易理解服务价值

### Phase 4：办案助手

目标：从咨询入口延伸到案件代理。

功能：

- 材料 OCR 和分类
- 证据目录
- 时间线
- 法条检索
- 类案检索
- 起诉状 / 答辩状 / 律师函草稿

验证指标：

- 办案准备时间是否下降
- 文书草稿可用率
- 证据遗漏率是否下降

## 12. 当前下一步

建议下一步不要立刻做完整系统，而是先做一个可验证的咨询助手原型。

优先任务：

1. 选定第一个咨询场景，例如劳动争议或民间借贷。
2. 写出该场景的标准问诊表。
3. 写出该场景的输出模板：咨询纪要、案情摘要、待补材料清单、初步法律分析。
4. 找 3-5 条真实或模拟咨询记录跑通。
5. 让律师判断：哪些内容可用，哪些内容必须改，哪些内容有风险。
6. 根据反馈迭代 playbook 和输出模板。

第一版最关键的不是功能多，而是证明：

> AI 能把混乱的客户咨询，稳定整理成律师愿意修改和使用的结构化工作底稿。
