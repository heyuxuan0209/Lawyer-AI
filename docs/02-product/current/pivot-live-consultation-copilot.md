# 产品方向修正：实时谈案 Copilot

更新时间：2026-05-19

## 1. 修正原因

前一阶段原型聚焦在：

```text
咨询记录 -> 咨询纪要 -> 案情结构化 -> 材料清单 -> 法律意见书草稿
```

这套方向本质是 **咨询后整理 / 交付件生成**。

但律师朋友实际要的是：

```text
语音转记
-> 咨询过程中实时抓重点
-> 过程中形成解决方案
-> 实时推送相关法律条款
```

因此，产品主线需要从“咨询后整理助手”修正为：

```text
律师实时谈案 Copilot
```

原有咨询纪要、案情结构化、材料清单、法律意见书草稿不废弃，但从主流程降级为 **咨询结束后的沉淀产物**。

## 2. 新产品定位

产品名称暂定：

```text
律师实时谈案助手
```

一句话定义：

> 在律师与客户咨询过程中，实时完成语音转记、案情要点提取、追问提醒、法条推送和方案草稿形成，咨询结束后自动沉淀为纪要、材料清单和法律意见书草稿。

核心价值：

- 减少律师手动记录成本。
- 防止咨询中漏问关键事实。
- 边聊边形成可见的解决方案。
- 实时提示法律条款和风险边界。
- 咨询结束后自动生成留痕和交付件。

## 3. 新核心工作流

```text
开始咨询
-> 实时语音转文字
-> 自动分段记录客户陈述
-> 实时识别案情要点
-> 实时提醒律师追问
-> 实时推送相关法律条款
-> 过程中形成初步方案
-> 律师确认/修正方案
-> 咨询结束
-> 自动生成咨询纪要、材料清单、意见书草稿
```

## 4. 产品主界面调整

仍采用三栏式，但含义调整。

### 4.1 左栏：咨询会话列表

一级对象仍是“咨询线程”。

分组建议：

- 正在咨询
- 待补材料
- 待出方案
- 已结束

卡片字段：

- 客户
- 案由
- 咨询状态
- 当前已识别争议
- 是否正在录音
- 是否需要律师回应

### 4.2 中栏：实时语音转记流

中栏成为核心区域，不再是会后产物流。

内容包括：

- 实时转写文本
- 客户发言分段
- 律师发言分段
- AI 自动标记的关键信息
- AI 对当前段落的理解
- 重要原话高亮

示例：

```text
客户：我 2023 年 3 月入职，公司上个月说我绩效不达标，让我不用来了。

AI 标记：
- 入职时间：2023 年 3 月
- 解除理由：绩效不达标
- 解除方式：口头通知
- 待追问：是否收到书面解除通知？
```

### 4.3 右栏：实时辅助面板

右栏不再优先放意见书，而是放咨询中律师最需要的辅助信息。

模块优先级：

1. 已识别案情要点
2. 建议追问
3. 相关法律条款
4. 初步方案草稿
5. 风险提示话术
6. 咨询结束后产物

右栏核心不是“展示结果”，而是 **辅助律师下一句话怎么问、怎么判断、怎么推进**。

## 5. MVP 边界

### 5.1 第一版必须做

先做演示型 MVP，不立即接真实语音模型。

必须展示：

- 点击“开始咨询”
- 中间模拟实时转写逐段出现
- 每段出现时，右侧同步更新：
  - 案情要点
  - 建议追问
  - 相关法律条款
  - 初步方案
  - 风险话术
- 点击“结束咨询”
- 自动生成咨询纪要、待补材料清单、方案摘要

### 5.2 第一版暂不做

- 真实语音识别
- 实时麦克风输入
- 法条数据库检索
- 类案检索
- 账号系统
- 客户门户
- 正式法律意见书导出
- 复杂多案由

### 5.3 为什么先模拟实时

因为当前最需要验证的是：

```text
律师是否需要“边听边提示”的工作台？
右侧推送的信息是否真的能帮他咨询？
法条推送和追问提醒会不会干扰咨询节奏？
```

真实语音识别可以后置。先验证交互和信息价值。

## 6. 劳动争议实时谈案示例

### 6.1 转写片段

```text
客户：我 2023 年 3 月入职一家科技公司，做运营，工资每月 12000。
```

右侧更新：

```text
案情要点：
- 入职时间：2023 年 3 月
- 岗位：运营
- 工资：12000 元/月

建议追问：
- 是否签订劳动合同？
- 工资是税前还是税后？
- 工资通过银行发放还是现金？
```

### 6.2 转写片段

```text
客户：上个月公司说我绩效不达标，让我直接不用来了。
```

右侧更新：

```text
案情要点：
- 解除理由：绩效不达标
- 解除方式：口头通知

建议追问：
- 是否收到书面解除通知？
- 是否签过绩效考核结果？
- 是否有绩效改进计划？
- 公司制度是否明确绩效不达标后果？

相关法律条款：
- 用人单位解除劳动合同需有合法依据和程序。
- 若违法解除，可能涉及赔偿金。
```

### 6.3 转写片段

```text
客户：我有劳动合同、工资流水，还有一些微信聊天记录，但是考勤记录不完整。
```

右侧更新：

```text
材料状态：
- 已有：劳动合同、工资流水、部分微信记录
- 缺失：解除通知、绩效材料、完整考勤、加班证明

初步方案：
- 先固定劳动关系和工资标准
- 核验解除依据和程序
- 补充绩效、解除、考勤和加班证据
- 判断是否走劳动仲裁
```

## 7. 与旧版本关系

旧版本保留，但定位调整：

| 旧模块 | 新定位 |
| --- | --- |
| 咨询纪要 | 咨询结束后自动沉淀 |
| 案情结构化 | 实时过程中持续更新 |
| 待补材料清单 | 实时提醒 + 结束后汇总 |
| 初步法律分析 | 右侧方案草稿持续更新 |
| 法律意见书草稿 | 付费咨询结束后的可选交付件 |
| 律师反馈按钮 | 继续保留，用于评估 AI 提示是否可用 |

## 8. 新版本规划

### v0.6-live-workspace-plan

当前版本，先完成整体规划，不开发。

产物：

- 产品方向修正文档
- 实时谈案 Copilot PRD
- 版本路线图
- 决策日志更新

### v0.7-live-static-prototype

做静态演示版：

- 开始咨询按钮
- 模拟实时转写
- 右侧实时更新案情要点、追问、法条、方案
- 结束咨询后生成沉淀产物

### v0.8-live-feedback

验证咨询中辅助是否有用：

- 每条追问建议可标记：有用 / 干扰 / 错误
- 每条法条推送可标记：相关 / 不相关 / 风险高
- 方案草稿可标记：可用 / 需修改 / 有风险

### v0.9-real-transcription-prototype

接入真实转写能力：

- 先支持上传音频转写
- 再考虑实时麦克风转写
- 保留手动粘贴转写文本作为 fallback

### v1.0-real-legal-retrieval

接入真实法律条款检索：

- 先做劳动争议高频条款包
- 再接法条库 / 检索 API
- 法条推送必须标注来源和核验状态

## 9. 给用户确认的问题

在进入 v0.7 开发前，需要确认：

1. 第一版实时原型是否仍以劳动争议为例？
2. 中栏转写是否采用“客户/律师分角色对话流”？
3. 右栏法条推送第一版是否先用“模拟法条卡”，不接真实检索？
4. 咨询结束后是否仍保留“生成纪要/材料清单/方案摘要”？
5. 是否暂时不做法律意见书，避免第一版显得过重？
