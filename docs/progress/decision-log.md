# Lawyer-AI 项目决策日志

本文档用于记录这个项目从 0 到 1 过程中的关键判断、取舍、踩坑、复盘和验证逻辑。

它不是普通需求文档，也不是开发日志，而是项目的“决策账本”。目标是把我们为什么做这个方向、为什么这样做、为什么不做其他方向、如何验证判断、事后如何反思，持续沉淀下来。

这份文档也会服务于未来的面试、融资、复盘和产品叙事。

## 使用方式

每当项目发生以下变化，都应在本文档补一条记录：

- 产品方向发生调整。
- MVP 范围发生收缩或扩张。
- 目标用户、场景、付费路径发生变化。
- 因用户访谈或演示反馈推翻了原判断。
- 技术路线、部署方式、数据策略、合规边界出现关键取舍。
- 某个功能被决定暂缓、删除或降级。

单条记录建议包含：

```text
日期：
决策：
背景：
为什么这样做：
为什么不做其他方案：
验证方式：
风险与待观察：
事后复盘：
```

## 当前项目核心判断

截至 2026-05-22，项目核心判断如下：

```text
Lawyer-AI 当前不是泛法律问答产品，也不是替代律师的法律意见生成器。

第一阶段聚焦律师客户咨询场景，尤其是劳动争议咨询。

产品主线是：
实时语音/对话转写
-> 案情要点结构化
-> 建议追问
-> 可能相关规则/法条卡
-> 方案草稿
-> 咨询结束后生成纪要、材料清单、方案摘要和模拟法律意见书格式样例

AI 的角色是律师谈案 Copilot，不是直接面向客户的法律顾问。
```

## 决策记录

### 2026-05-19：第一阶段先做客户咨询，而不是文书、资讯、营销或完整办案

决策：

优先切入客户咨询场景，产品定位为“客户咨询助手 / 谈案助手”。

背景：

最初讨论过 `claude-for-legal`、`AlphaGPT` 等法律 AI 产品能否借鉴，也列过律师日常工作中的多个场景。最终认为客户咨询更适合作为第一切入点。

为什么这样做：

- 客户咨询离律师成交最近，商业价值更直接。
- 输入天然存在，包括电话、线上平台、面对面沟通、微信记录、会议纪要、客户材料和律师备注。
- 风险边界相对可控，AI 只做律师复核草稿，不直接给客户最终法律意见。
- 后续可自然延展到法律意见书、律师函、起诉状、证据目录和案件工作台。

为什么不先做其他方向：

- 文书生成容易陷入模板竞争，且需要更强事实核验和责任边界。
- 法律资讯/检索离付费转化较远，难以证明律师愿意持续使用。
- 完整办案系统范围过大，第一阶段很难验证一个清晰价值点。

验证方式：

- 先用劳动争议场景做 MVP。
- 给律师朋友演示，重点验证其真实咨询中是否需要“转写、抓重点、追问、规则推送、方案形成”。

风险与待观察：

- 律师是否愿意在真实咨询现场打开一个工具。
- AI 实时提示是否会干扰律师谈案节奏。
- 客户隐私和数据合规边界后续必须明确。

事后复盘：

- 待律师演示反馈后补充。

### 2026-05-19：MVP 先做劳动争议

决策：

第一个 MVP 选择劳动争议付费咨询。

为什么这样做：

- 劳动争议高频，律师朋友更容易判断价值。
- 事实结构相对清楚，适合做结构化字段。
- 材料类型明确，如劳动合同、工资流水、社保记录、解除通知、考勤记录、聊天记录。
- 容易形成标准问诊表、材料清单、方案摘要和法律意见书格式样例。

为什么不先做其他案由：

- 婚姻家事、刑事、公司争议等场景情绪、事实和风险边界更复杂。
- 第一版需要快速验证产品机制，而不是覆盖复杂法域。
- 劳动争议更适合做“追问清单 + 证据清单 + 处理路径”的 MVP。

验证方式：

- 用“绩效解除争议”模拟咨询流。
- 看律师是否认可 AI 抓取的入职时间、工资、解除方式、绩效流程、证据材料和仲裁路径。

风险与待观察：

- 劳动争议是否过窄，能否代表其他法律咨询场景。
- 不同地区、不同律师对劳动争议处理路径可能不同。

事后复盘：

- 待至少 3 位律师反馈后复盘是否继续深挖劳动争议，还是扩到合同/婚姻/公司争议。

### 2026-05-19：第一版先做咨询后整理，不做实时语音

决策：

最初第一版设计为支持粘贴咨询记录、录音转写文本、微信聊天记录和律师备注，暂不做实时录音转写。

当时为什么这样做：

- 核心验证点看起来是输出质量，而不是实时能力。
- 咨询后整理更容易做出 MVP，风险更低。
- 可以更快用固定脱敏样本验证产物链路。

后来发生了什么：

用户指出律师真正想要的是“给他做语音转记，过程中形成方案，还可以推送法律条款”。这推翻了“会后整理是主价值”的初始判断。

复盘：

- 这条是一次关键方向修正。
- 原判断低估了咨询现场的认知负荷，高估了会后文书生成的吸引力。
- 会后产物仍有价值，但不应作为第一屏主卖点。

当前处理：

- 咨询纪要、材料清单、方案摘要、模拟法律意见书格式样例保留。
- 它们被降级为“咨询结束后的沉淀产物”。

### 2026-05-19：采用三栏式工作台

决策：

参考 `Mall Operation Agent` 的 `/workspace` 信息组织方式，法律咨询助手也采用三栏式工作台。

映射关系：

```text
经营事项线程 -> 咨询线程
会商消息流 -> 咨询转写/谈案过程
证据账本/决策栏 -> 案情要点/追问/法条卡/方案草稿/会后产物
```

为什么这样做：

- 一级对象是“咨询线程”，不是后台菜单。
- 中间区域适合呈现客户和律师对话流。
- 右侧适合放 AI 实时辅助，符合律师边听边看的使用方式。
- 左侧保留其他咨询线程，可以表达未来工作台形态。

后续调整：

- 其他咨询线程保留但置灰，避免律师误以为可以点击切换。
- 反馈控件从实时区域移出，改成“反馈模式”下的会后回看评价。

风险与待观察：

- 律师在真实咨询中是否愿意看右侧辅助信息。
- 三栏信息密度在小屏或移动端是否过高。

### 2026-05-19：方向修正，从咨询后整理转为实时谈案 Copilot

决策：

主产品方向从“咨询后整理 / 交付件生成”调整为“实时谈案 Copilot”。

背景：

用户指出律师的真实需求是：

```text
语音转记
咨询过程中形成方案
过程中推送法律条款
```

为什么这样做：

- 律师在咨询现场需要同时听客户、安抚客户、追问事实、组织法律判断。
- AI 如果只在结束后生成文档，错过了最关键的认知负荷点。
- 实时案情要点、建议追问、法条卡和方案草稿更贴近律师谈案过程。

为什么不继续主打法律意见书：

- 法律意见书风险边界更重，且需要完整事实和证据核验。
- 律师第一眼更关心的是咨询过程能不能被辅助，而不是 AI 能不能替他写正式意见。
- 法律意见书可作为格式样例保留，用于验证交付结构，但不应成为 MVP 核心。

新主线：

```text
语音转记
-> 实时案情要点提取
-> 实时追问建议
-> 实时法律条款/规则卡推送
-> 过程中形成方案草稿
-> 咨询结束后沉淀纪要、材料清单、方案摘要
```

验证方式：

- 做劳动争议单页静态原型。
- 用模拟客户/律师分角色对话验证咨询现场辅助价值。
- 暂不接真实 ASR 和真实法条检索。

风险与待观察：

- 实时提示可能打断律师节奏。
- 法条卡如果过度确定，容易造成法律风险。
- 真实语音转写质量会直接影响后续结构化质量。

### 2026-05-19：第一版原型使用模拟流，不接真实麦克风和真实法条库

决策：

第一版部署给律师看的版本使用模拟实时咨询流、模拟法条卡和模拟法律意见书格式样例。

为什么这样做：

- 当前阶段要验证工作流价值，而不是技术可用性。
- 真实 ASR、法条检索、知识库和合规审查都会显著增加开发成本。
- 用模拟流可以快速让律师判断“这个东西如果是真的，是否值得继续做”。

为什么不立即接真实能力：

- 真实转写会引入噪音，第一轮反而不利于判断产品形态。
- 法条库接入前必须设计引用、版本、适用前提和责任边界。
- 先验证律师是否认可“实时 Copilot”的形态，再决定技术投入。

风险与待观察：

- 模拟原型可能让律师误以为已经具备真实能力。
- 需要在页面明确提示“模拟演示、未经律师审核不可发送客户”。

当前处理：

- 页面顶部增加模拟演示提示。
- 法条卡使用“可能相关规则”“适用前提”“需律师确认”等克制表达。
- 法律意见书明确为格式样例。

### 2026-05-19：反馈模式改为会后回看，而不是实时打分

决策：

律师反馈控件默认隐藏，打开“反馈模式”后出现；反馈方式从实时打分改为咨询结束后回看评价。

背景：

原设计在右侧建议追问、模拟法条卡、方案草稿下直接放“有用/干扰/错误”等按钮。用户指出对话流太快，反馈模式跟随变化时来不及反应。

为什么这样做：

- 演示时律师应先完整理解产品价值，而不是被迫边看边点。
- 会后回看更符合真实评审节奏。
- 反馈控件不应污染正式产品视角。

为什么不保留实时打分：

- 实时打分会和谈案场景冲突。
- 律师容易把反馈按钮误解为业务操作。
- 快速变化的右侧内容会导致反馈对象不清晰。

验证方式：

- 给律师演示时先关闭反馈模式。
- 完整看完咨询后，再打开反馈模式，让律师评价追问、法条卡、方案草稿。

待观察：

- 会后评价是否足够准确。
- 是否需要支持对某一条具体追问或某一张具体法条卡逐条评论。

### 2026-05-20：部署优先选择 Vercel，Zeabur 因套餐限制暂缓

决策：

第一版线上演示部署到 Vercel。

背景：

用户有 Zeabur 账号，最初尝试部署到 Zeabur。但 Zeabur CLI 创建新服务时返回 `REQUIRE_PAID_PLAN`，提示当前 `cgk1` 区域需要 Developer Plan。

为什么改用 Vercel：

- 当前原型是纯静态页面，Vercel 可以快速部署。
- 用户已经能提供 Vercel Token。
- 第一阶段目标是拿到可分享链接，而不是确定长期生产基础设施。

为什么不继续卡在 Zeabur：

- 套餐/区域限制会拖慢验证。
- 项目还处于演示验证阶段，没有必要为了部署平台投入过多时间。

结果：

线上链接：

```text
https://prototype-pi-weld.vercel.app
```

后续处理：

- GitHub 仓库已连接 Vercel。
- 后续通过 Git push 触发自动部署。

风险与待观察：

- Vercel 在中国大陆访问可能不稳定。
- 如果律师反馈打不开或太慢，应迁移到更适合大陆访问的 EdgeOne Pages、国内 CDN 或自有备案域名。

### 2026-05-20：绑定 GitHub 自动部署，保持同一个演示链接

决策：

将本地仓库推送到 GitHub，并连接 Vercel 自动部署。

背景：

用户希望后续高频迭代时继续使用同一个链接，而不是每次生成新链接。

为什么这样做：

- 每次 `git push` 后 Vercel 自动部署，降低发布成本。
- 演示链接保持稳定，方便发给律师反复查看。
- 项目迭代记录和部署记录形成闭环。

实现：

- GitHub 仓库：

```text
https://github.com/heyuxuan0209/Lawyer-AI
```

- Vercel 项目：

```text
prototype
```

- 线上链接：

```text
https://prototype-pi-weld.vercel.app
```

- 新增根目录 `vercel.json`，将 Vercel 根路径路由到 `prototype/index.html`，避免必须在 UI 中配置 Root Directory。

验证方式：

- push 后线上链接返回 `HTTP/2 200`。

风险与待观察：

- 根路径 rewrite 适合当前静态原型，但未来如果项目升级为正式前端应用，需要重新整理目录和部署配置。

## 待验证假设

### H1：律师愿意在咨询现场使用实时辅助工具

验证问题：

- 如果这个工具真实接入麦克风，你是否愿意在客户咨询时打开？
- 它会帮助你，还是打断你？
- 你更愿意它在电脑屏幕、手机、平板还是会议软件侧边栏出现？

### H2：建议追问比会后文书更接近第一价值点

验证问题：

- 哪些追问是你真实会问的？
- 哪些追问没必要，甚至会暴露不专业？
- 你希望追问显示 1 条、3 条还是完整清单？

### H3：法条卡的价值在于“判断框架”，不是堆法条全文

验证问题：

- 你希望看到具体法条号，还是先看到规则主题？
- “为什么推送 / 适用前提 / 需核验”这种卡片结构是否合适？
- 法条卡什么时候出现最不打扰？

### H4：会后产物中材料清单和方案摘要可能比法律意见书更早产生价值

验证问题：

- 咨询结束后，如果只能保留一个产物，你选咨询纪要、材料清单、方案摘要还是法律意见书？
- 哪些内容可以直接发客户？
- 哪些内容必须只作为律师内部草稿？
